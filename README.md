# RAG Agent Backend

Независимый backend-сервис агента и RAG для LLM-системы. Проект рассчитан на локальную разработку и использует Ollama как внешний model backend через отдельный интеграционный слой.

## Что нужно установить перед запуском

Для комфортного локального запуска проекта на Ubuntu/Debian желательно заранее установить базовые системные пакеты и инструменты.

Нужно установить:

- Python 3.11+
- `python3-venv`
- `python3-pip`
- `build-essential`
- `python3-dev`
- `curl`
- Ollama

Пример установки для Ubuntu/Debian:

```bash
sudo apt update
sudo apt install -y python3 python3-venv python3-pip build-essential python3-dev curl
```

Ollama устанавливается отдельно. После установки нужно убедиться, что сервис доступен и что загружены необходимые модели.

## Что внутри

- FastAPI backend
- отдельный `integrations/ollama` слой
- базовая агентная orchestration-логика
- простой RAG pipeline: ingestion -> chunking -> embeddings -> retrieval
- минимальный web UI для локальной проверки

## Ограничения MVP

- сейчас поддерживается только Ollama
- для чата и embeddings нужны доступные модели в Ollama
- проект не зависит от Ansible-стенда и не требует его для локального запуска

## Конфигурация

Основной конфиг приложения теперь хранится в одном файле:

- `config.toml` — главный редактируемый конфиг сервиса

Через него настраиваются:

- параметры HTTP-сервера
- адрес и модели Ollama
- пути к данным
- параметры chunking и retrieval
- включение встроенного тестового UI

Если нужно использовать другой конфиг, можно указать путь через переменную окружения:

```bash
RAG_AGENT_CONFIG=/path/to/config.toml
```

`.env` не является обязательным для работы сервиса и не содержит основную логику конфигурации. Файл `.env.example` оставлен только как пример env-переопределений.

## Быстрый старт

1. Создать окружение и установить зависимости:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -e .
```

2. При необходимости отредактировать `config.toml`.

3. Убедиться, что Ollama доступна и модели загружены:

```bash
ollama pull llama3.2:1b
ollama pull nomic-embed-text
```

4. Запустить сервис:

```bash
python3 -m rag_agent --reload
```

5. Открыть UI:

`http://localhost:8000/`

## Запуск тестов

Запуск всех тестов:

## Основные endpoint'ы

- `GET /health` — проверка доступности сервиса
- `POST /api/v1/chat` — генерация ответа через agent-логику
- `POST /api/v1/rag/search` — retrieval без генерации
- `POST /api/v1/documents/upload` — загрузка документов в индекс

## Типовой сценарий работы

1. Пользователь отправляет запрос из веб-интерфейса.
2. Backend при необходимости выполняет retrieval по загруженным документам.
3. Backend передает запрос и релевантный контекст в Ollama.
4. Backend возвращает готовый ответ и найденные фрагменты контекста.

## Формат retrieval-контекста

Для endpoint'ов, связанных с RAG, backend возвращает найденный контекст в виде списка объектов:

```json
{
  "text": "Фрагмент текста",
  "source": "Из какого файла взят фрагмент",
  "chunk_id": "README.md:0"
}
```

Это позволяет внешнему вебу не только показывать текст контекста, но и знать, из какого файла он был получен.

## Интеграция с внешним вебом

Встроенный `frontend/index.html` в этом проекте является только минимальным тестовым UI для локальной проверки backend-логики.

Он не является обязательной частью production-схемы и не должен определять архитектуру системы.

Основной способ интеграции проекта с любым веб-интерфейсом — HTTP API backend-сервиса.

Если в системе уже есть свой frontend, встроенный `frontend/` можно не использовать вообще.

Текущий `frontend/index.html` нужен только для локального тестирования и smoke test. Он подключен внутри backend как вспомогательный интерфейс, но не является обязательной частью интеграции.

При использовании внешнего веба достаточно обращаться к backend по HTTP API. В этом сценарии встроенный UI можно просто игнорировать без изменений backend-логики.

Любой внешний веб может быть подключен к этому backend, если он умеет вызывать следующие endpoint'ы:

- `GET /health`
- `POST /api/v1/chat`
- `POST /api/v1/rag/search`
- `POST /api/v1/documents/upload`

Для endpoint'ов, использующих retrieval, поле `contexts` возвращается не как список строк, а как список объектов с данными о найденном фрагменте:

- `text` — текст найденного чанка
- `source` — источник, обычно имя файла
- `chunk_id` — идентификатор чанка в индексе

Типовая схема интеграции:

`external web -> this backend -> Ollama`

Что важно:
- внешний веб не обязан использовать встроенный `frontend/`
- backend не зависит от конкретного frontend-стека
- frontend можно разрабатывать и разворачивать отдельно
- для подключения нового веба обычно не нужно менять backend-код, если достаточно существующего API

Пример запроса к чату:

```bash
curl -X POST http://localhost:8000/api/v1/chat \
  -H "Content-Type: application/json" \
  -d '{
    "message": "О чем этот проект?",
    "use_rag": true,
    "top_k": 3
  }'
```

Пример ответа:
```json
{
  "answer": "Краткий ответ модели",
  "contexts": [
    {
      "text": "Фрагмент релевантного контекста 1",
      "source": "README.md",
      "chunk_id": "README.md:0"
    },
    {
      "text": "Фрагмент релевантного контекста 2",
      "source": "README.md",
      "chunk_id": "README.md:1"
    }
  ]
}
```

### Если веб уже существует в системе

Если в системе уже поднят свой веб-интерфейс, обычно не нужно менять backend-код этого проекта.

Нужно сделать следующее:

1. Поднять этот backend-сервис отдельно и убедиться, что он доступен по HTTP, например:
   - `http://localhost:8000`
   - `http://agent-backend:8000`
   - `http://<host>:8000`

2. Во внешнем вебе указать адрес этого backend как отдельного API-сервиса.

3. Настроить вызовы endpoint'ов backend из веба:
   - `POST /api/v1/chat` — отправка пользовательского запроса
   - `POST /api/v1/rag/search` — retrieval без генерации, если нужен отдельный поиск
   - `POST /api/v1/documents/upload` — загрузка документов в RAG
   - `GET /health` — проверка доступности сервиса

4. Если веб показывает найденный контекст, нужно учитывать, что поле `contexts` возвращается как список объектов, а не строк.

Пример объекта контекста:

```json
{
  "text": "Фрагмент документа",
  "source": "README.md",
  "chunk_id": "README.md:0"
}
```

Это значит, что веб-клиент должен уметь:

- брать текст из `text`
- при необходимости показывать источник из `source`
- при необходимости использовать `chunk_id` для отладки или отображения

Встроенный `frontend/` в этом репозитории можно не использовать, если внешний веб уже существует.

Он нужен только для локального тестирования backend-сервиса.

Пример логики интеграции внешнего веба с backend:

- пользователь вводит вопрос в существующем вебе
- веб отправляет запрос в `POST /api/v1/chat`
- backend при необходимости делает retrieval и вызывает Ollama
- backend возвращает готовый ответ и список найденных контекстов
- веб отображает ответ пользователю

Упрощенная схема:

`existing web -> this backend -> Ollama`

## Настройка подключения к Ollama

В текущем MVP backend работает только с Ollama.

Ollama рассматривается как внешний model backend и должна быть доступна по HTTP API. Внутренняя логика backend не должна обращаться к Ollama напрямую вне интеграционного слоя.

Подключение настраивается в `config.toml`:

```toml
[ollama]
base_url = "http://localhost:11434"
chat_model = "llama3.2:1b"
embedding_model = "nomic-embed-text"
timeout = 60.0
```

Если Ollama запущена не локально, достаточно изменить `base_url`, например:

```toml
[ollama]
base_url = "http://192.168.1.50:11434"
chat_model = "llama3.2:1b"
embedding_model = "nomic-embed-text"
timeout = 60.0
```

Если нужно сменить модель, меняются только значения в конфиге, без изменения бизнес-логики backend.

Точки, связанные с конфигурацией и интеграцией:

- конфиг приложения: `src/rag_agent/core/config.py`
- клиент интеграции с Ollama: `src/rag_agent/integrations/ollama/client.py`

Остальные модули проекта должны работать с LLM через сервисный и интеграционный слой, а не через прямые HTTP-вызовы к Ollama.

## Что менять под свой стенд

### Если используется внешний веб

Если в системе уже есть свой веб-интерфейс, ориентируйтесь на раздел `Интеграция с внешним вебом`.

На практике это означает:
- backend поднимается как отдельный HTTP API-сервис
- внешний веб обращается к `POST /api/v1/chat`, `POST /api/v1/rag/search`, `POST /api/v1/documents/upload`
- встроенный `frontend/` можно не использовать

Если внешний веб использует retrieval-ответы, он должен уметь обрабатывать `contexts` как список объектов с полями `text`, `source` и `chunk_id`.

### Если нужно заменить встроенный тестовый UI

Если требуется раздавать UI прямо из этого backend, можно заменить содержимое папки `frontend/`.

Текущая раздача тестового UI подключена в файле:

- `src/rag_agent/main.py`

Сейчас backend:
- отдает `frontend/index.html` на `/`
- монтирует `frontend/` как статические файлы

### Если Ollama вынесена в отдельный сервис

В этом случае нужно изменить только секцию `[ollama]` в `config.toml`:

- `base_url`
- при необходимости `chat_model`
- при необходимости `embedding_model`

### Если меняются локальные пути данных

Через `config.toml` можно также настроить секцию `[data]`:
- `dir`
- `documents_dir`
- `index_path`

Это позволяет хранить документы и индексы в нужных каталогах без изменения кода.

### Если нужно настроить чувствительность retrieval

Через секцию `[rag]` в `config.toml` можно управлять параметрами retrieval:

- `default_top_k` — сколько кандидатов брать из retrieval
- `min_retrieval_score` — минимальный score, ниже которого найденный chunk не будет использоваться

Пример:

```toml
[rag]
default_top_k = 4
min_retrieval_score = 0.2
```

Если retrieval возвращает слишком много нерелевантного контекста, имеет смысл увеличить `MIN_RETRIEVAL_SCORE`.

Если retrieval, наоборот, слишком часто не находит контекст, порог можно уменьшить.

## Типичные проблемы

### Не создается виртуальное окружение

Если команда

```bash
python3 -m venv .venv
```

не работает, чаще всего в системе не установлен пакет `python3-venv`.

Для Ubuntu/Debian нужно проверить, что он установлен:

```bash
sudo apt install -y python3-venv
```

### Backend запускается, но чат не отвечает

Чаще всего это означает, что Ollama не запущена, недоступна по сети или указан неверный `base_url` в секции `[ollama]`.

Нужно проверить:

- что Ollama действительно запущена
- что адрес в `config.toml` совпадает с реальным адресом сервиса
- что backend может достучаться до Ollama по HTTP API

### Ollama доступна, но генерация или embeddings не работают

Обычно причина в том, что нужные модели не загружены.

Нужно проверить, что доступны:

- модель для чата, например `llama3.2:1b`
- embedding-модель, например `nomic-embed-text`

При необходимости их нужно загрузить:

```bash
ollama pull llama3.2:1b
ollama pull nomic-embed-text
```

### Retrieval ничего не находит

Возможные причины:

- документы еще не были загружены в индекс
- `MIN_RETRIEVAL_SCORE` слишком высокий
- запрос и загруженные документы действительно слабо связаны по смыслу

Что проверить:

- загружались ли документы через `POST /api/v1/documents/upload`
- существует ли локальный индекс
- не слишком ли высокий порог `MIN_RETRIEVAL_SCORE`

### Сервис не стартует из-за занятого порта

Если порт `8000` уже используется, `uvicorn` не сможет запуститься.

В этом случае можно:

- освободить занятый порт
- или изменить `port` в секции `[app]` файла `config.toml`

Пример:

```toml
[app]
port = 8001
```

### После запуска не открывается локальный UI

Нужно проверить:

- что backend действительно запущен
- что открыт адрес `http://localhost:8000/`
- что порт совпадает со значением `port` в секции `[app]`

Если порт был изменен, адрес тоже нужно открывать с новым портом.
