# AGENTS

## Назначение

Этот файл задает правила для AI-ассистентов, которые помогают разрабатывать данный проект.

Проект представляет собой отдельный backend-сервис агента и RAG для LLM-системы.

---

## 1. Общий контекст

Существующий командный стенд с LLM уже есть, но этот репозиторий разрабатывается **отдельно от него**.

Нужно исходить из того, что:
- основной инфраструктурный стенд существует отдельно;
- в нем уже может быть поднят model backend;
- в нем уже может существовать веб;
- наша задача — создать независимый сервис, который потом можно интегрировать в систему.

Не нужно предлагать решения, которые требуют глубокой переделки:
- Ansible-ролей;
- логики существующего стенда;
- внутренней структуры инфраструктурного репозитория.

---

## 2. Главные архитектурные правила

### 2.1 Не связывать проект с Ansible
Нельзя строить внутреннюю логику проекта вокруг:
- inventory;
- group_vars;
- ansible roles;
- конкретного сценария деплоя.

Проект должен быть самодостаточным приложением.

### 2.2 Работать с LLM только через интеграционный слой
В текущей версии проекта поддерживается **только Ollama**.

Однако:
- нельзя размазывать прямые HTTP-вызовы Ollama по разным модулям;
- нельзя делать так, чтобы agent, rag и api знали детали payload формата Ollama.

Нужно держать отдельный слой интеграции, например:
- `OllamaClient`
- `OllamaAdapter`
- `LLMService`

### 2.3 Core-логика должна быть чистой
Логика:
- агента,
- retrieval,
- chunking,
- pipeline обработки документов,
- orchestration

должна быть отделена от:
- веб-слоя;
- конфигурации окружения;
- прямой работы с модельным API.

### 2.4 Не делать лишнюю универсальность
Сейчас не нужно:
- проектировать поддержку 5 разных провайдеров;
- строить слишком абстрактный framework;
- добавлять сложные паттерны без необходимости.

Нужно делать:
- минимально достаточную архитектуру;
- читаемый код;
- практичный MVP.

---

## 3. Ограничения среды разработки

Нужно учитывать:

- разработка идет на Linux-ноутбуке;
- ноутбук слабый по графике;
- тестирование моделей идет на Windows-ПК с RTX 4060;
- возможен запуск тестовой среды через WSL;
- GPU-контейнеры для разработки не планируются;
- приложение не должно зависеть от GPU-specific логики.

Следствие:
- решения должны быть легкими;
- локальная разработка должна быть возможна без тяжелых требований;
- модельная часть должна быть максимально вынесена наружу.

---

## 4. Правила проектирования

### 4.1 Делать сначала MVP
Приоритет:
1. рабочий backend;
2. рабочий чат с моделью;
3. рабочий RAG;
4. удобный API;
5. минимальный web UI для локального тестирования.

### 4.2 Предпочитать простые зависимости
Выбирать простые и понятные инструменты.

Предпочтительно:
- Python
- FastAPI
- Pydantic
- requests/httpx
- простой vector store или легкая локальная реализация на старте

Не нужно без необходимости тащить тяжелый стек.

### 4.3 Код должен быть переносимым
Нельзя жестко завязываться на:
- Linux-only пути;
- специфичные shell-команды без альтернативы;
- окружение конкретной машины;
- ручные действия, которые трудно повторить.

### 4.4 Все важное должно конфигурироваться
Через `.env` или конфиг должны задаваться:
- base URL Ollama;
- название модели;
- embedding model;
- пути к данным;
- параметры индексации;
- параметры сервера.

---

## 5. Правила по структуре кода

Желательно придерживаться такого разделения:

- `api/` — роуты и схемы запросов/ответов
- `agent/` — логика агента
- `rag/` — ingestion, chunking, retrieval
- `integrations/ollama/` — вся работа с Ollama API
- `storage/` — файлы, индексы, локальные БД
- `core/` — общие интерфейсы и сервисы
- `tests/` — тесты

Не смешивать:
- API-роуты с бизнес-логикой;
- бизнес-логику с HTTP-вызовами к Ollama;
- загрузку документов с retrieval-логикой.

---

## 6. Правила для AI при генерации кода

AI-ассистент должен:

- писать код небольшими понятными модулями;
- избегать ненужного усложнения;
- сохранять читаемость;
- по возможности давать MVP-реализацию раньше “идеальной архитектуры”;
- не предлагать переписывать все с нуля без необходимости;
- не навязывать лишние abstractions.

AI-ассистент не должен:

- делать сверхсложную enterprise-архитектуру;
- вводить десятки слоев абстракции ради будущего;
- размазывать конфигурацию по случайным местам;
- делать hidden magic в коде.

---

## 7. Правила по Ollama

Считать, что:
- Ollama обязательна для работы проекта;
- это должно быть отражено в README;
- модельный backend доступен по API;
- проект работает через Ollama как через внешний сервис.

Нужно избегать:
- прямой зависимости всех модулей от формата Ollama API;
- смешивания prompt-логики и HTTP-клиента;
- вшивания model-specific деталей в agent core без нужды.

---

## 8. Правила по вебу

Целевой production/web-интерфейс заранее неизвестен.

Поэтому:
- backend должен предоставлять понятный API;
- локальный web UI может быть минимальным;
- UI не должен определять архитектуру проекта.

Считать web UI:
- инструментом тестирования и демонстрации;
- а не источником основной бизнес-логики.

---

## 9. Правила по Git

В репозиторий должны попадать:
- исходный код;
- инструкции запуска;
- примеры конфигов;
- тесты;
- минимальный web UI, если он нужен для проекта.

В `.gitignore` должны попадать:
- `.env`
- `.venv/`
- `__pycache__/`
- кэши
- логи
- базы
- индексы
- загруженные локально документы
- временные файлы
- IDE-служебные файлы

---

## 10. Как AI должен помогать в этом проекте

AI должен помогать:
- проектировать MVP;
- предлагать минимальную рабочую архитектуру;
- писать backend-код;
- писать код интеграции с Ollama;
- писать RAG pipeline;
- писать простой тестовый web UI;
- улучшать структуру проекта постепенно.

AI должен отдавать приоритет:
- практичности;
- скорости реализации;
- простоте поддержки;
- переносимости;
- ясному разделению ответственности.

---

## 11. Ожидаемый результат

Нужно прийти к проекту, который:
- легко запускается локально;
- не зависит от командного Ansible-репозитория;
- работает через Ollama;
- содержит агента и RAG;
- может быть потом встроен в более крупную систему.
