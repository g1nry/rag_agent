# PROJECT BRIEF

## 1. Что это за проект

Мы разрабатываем отдельный сервис **агента и RAG** для LLM-системы.

Проект создается **как независимое приложение**, которое потом можно встроить в существующий стенд с LLM или в другую совместимую систему.

Текущий командный репозиторий с Ansible и инфраструктурой используется **только как источник контекста**:
- чтобы понимать, как устроен стенд;
- какие сервисы уже подняты;
- где находится Ollama;
- какие есть ограничения по окружению.

Этот проект **не должен менять бизнес-логику существующего стенда** и **не должен зависеть от внутренней логики Ansible**.

---

## 2. Основная идея

Нужно разработать:
- backend-сервис агента;
- backend-сервис RAG;
- интеграцию с Ollama;
- простой API для дальнейшего подключения веба.

Проект ориентирован на сценарий, где:
- модельный backend уже существует;
- веб может быть уже установлен отдельно;
- наша задача — добавить поверх этого собственную логику агента и retrieval.

---

## 3. Архитектурные принципы

### 3.1 Независимость от инфраструктуры
Наш сервис:
- не должен зависеть от Ansible-ролей;
- не должен зависеть от inventory, group vars и логики деплоя;
- не должен требовать изменений в существующем инфраструктурном коде для своей внутренней работы.

### 3.2 Взаимодействие с моделью только по API
Наш сервис должен работать с LLM только через внешний API.

Для текущего этапа допустимо делать реализацию **только под Ollama**.

При этом Ollama должна восприниматься как:
- внешний inference backend;
- поставщик generation/embeddings;
- не центральная часть архитектуры проекта.

### 3.3 Изоляция Ollama-логики
Нельзя размазывать прямые вызовы Ollama API по всему проекту.

Нужно держать отдельный интеграционный слой, например:
- `ollama_client`
- `ollama_adapter`
- `services/llm`

Вся остальная логика должна обращаться к этому слою, а не напрямую к HTTP API Ollama.

### 3.4 Независимость от конкретного веба
Целевой веб в системе уже существует, но его реализация нам неизвестна.

Поэтому нужно проектировать интеграцию **через внешний API-контракт**, а не через привязку к конкретному UI.

Идея:
- любой веб должен иметь возможность подключиться к нашему backend;
- наш backend не должен предполагать конкретный frontend-стек.

---

## 4. Практические ограничения

### 4.1 Среда разработки
Разработка ведется:
- на Linux-ноутбуке;
- ноутбук слабый по графике.

### 4.2 Среда тестирования моделей
Тестирование моделей выполняется:
- на основном ПК с Windows;
- GPU: RTX 4060.

### 4.3 WSL
Допускается использование WSL для тестовой среды на основном ПК.

На текущем этапе:
- не планируются GPU-контейнеры для самой разработки;
- не нужно специально проектировать container GPU runtime;
- достаточно, чтобы WSL видел CUDA и мог работать с модельным backend.

### 4.4 Требование к приложению
Приложение должно быть **GPU-agnostic**:
- оно не должно содержать бизнес-логику, завязанную на конкретный GPU runtime;
- все детали исполнения модели должны оставаться на стороне внешней среды.

---

## 5. Локальная разработка

Проект должен полноценно разрабатываться **локально**, отдельно от основного стенда.

Локальная схема может быть такой:

`test web ui -> backend (agent + rag) -> local ollama`

Дополнительно:
- локальная база / векторное хранилище;
- локальная папка с документами;
- локальные тестовые конфиги.

Локальная разработка нужна для того, чтобы:
- не зависеть от командного стенда;
- спокойно проектировать архитектуру;
- быстро проверять агентную и RAG-логику;
- потом переносить готовое решение в основную систему.

---

## 6. Текущее решение по стеку

На текущем этапе проект можно делать **только под Ollama**.

Это допустимо и упрощает реализацию.

Ограничение должно быть явно зафиксировано в README:
- для работы нужен Ollama;
- нужен доступный endpoint;
- нужны нужные модели;
- при необходимости нужны embedding-модели.

При этом даже при поддержке только Ollama нужно сохранить:
- чистую структуру проекта;
- отдельный слой интеграции;
- отдельную агентную логику;
- отдельную RAG-логику.

---

## 7. Что должно быть в MVP

MVP должен включать:

### Агент
- прием запроса пользователя;
- вызов LLM через Ollama;
- базовую orchestration-логику;
- подготовку ответа;
- возможность расширения tool-calling позже.

### RAG
- загрузку документов;
- разбиение на чанки;
- создание embeddings;
- индексирование;
- retrieval по запросу;
- подмешивание релевантного контекста в generation.

### API
- endpoint для чата;
- endpoint для RAG-запросов;
- endpoint для загрузки документов;
- health-check.

### Простой тестовый UI
- минимальный интерфейс для локального тестирования;
- не production frontend;
- нужен только для удобной проверки backend-логики.

## 9. Git и локальные файлы

В репозиторий должен попадать код проекта.

В .gitignore должны уходить:

- .venv/
- __pycache__/
- .env
- локальные базы
- логи
- кэш
- индексы векторного хранилища
- временные файлы
- IDE-файлы

Но не должны игнорироваться:

- основной backend-код;
- тестовый web UI, если он является частью проекта;
- конфиги запуска, которые нужны команде.
## 10. Главный принцип проекта

Этот проект — не часть Ansible-логики и не часть Ollama, а самостоятельный прикладной слой над существующим model backend.

Кратко:

- инфраструктура существует отдельно;
- Ollama существует отдельно;
- веб может существовать отдельно;
- наш проект добавляет агентную и RAG-логику как независимый сервис.
